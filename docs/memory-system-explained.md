# Research Companion：一个模拟人类记忆层级的Agent记忆系统

## 为什么需要记忆系统

受限于上下文窗口的限制，我们不能每次都把以前所有的聊天记录全部作为prompt输入给大模型。但人的"上下文"同样是有限的，却能完成长期的、连续的复杂任务。这启发我们思考：人是如何管理记忆的？能否借鉴这套机制为agent设计一个记忆系统？

人的记忆大致可以分为三层。第一层是**核心记忆**——我的名字是什么、我的身份是什么、我最近在做什么——这些信息始终存在于我们的"上下文"中，随时可用。第二层是**近期记忆**，也就是最近发生的事情。近期记忆又可以细分为两种：一种是刚刚发生的，比如面前的人刚才说了什么，它就在当前上下文中；另一种是发生了一会儿的，上下文中只留有模糊的印象，但一经提起就能完整回忆起来，比如中午吃了什么。第三层是**长期记忆**，平时不在上下文中，但如果被关键词触发，可以大致回忆起来，比如去年暑假去了哪里旅游。

根据这个思路，我为科研场景设计了一个模拟人类记忆层级的agent记忆系统——**Research Companion**。核心想法是把agent定位成一个长期的科研合作者（用户可以给它起个名字，此后每次叫它的名字就能触发这个skill），在科研过程中随时与它讨论想法、梳理思路。

## 三层记忆架构

### L1：核心记忆（Working Memory）

对应人的核心记忆，**每次对话都会完整加载**。L1包含五个文件：

| 文件 | 内容 |
|------|------|
| `project_profile.md` | 项目概况：课题、研究问题、方法、数据、项目结构、当前阶段、长短期目标 |
| `active_directions.md` | 当前研究方向及其状态（探索中/计划中/进行中/暂停），包括已完成和已放弃的方向 |
| `key_decisions.md` | 关键决策记录：上下文、决策内容、理由、曾考虑的替代方案 |
| `researcher_profile.md` | 研究者画像：用户的专长、当前兴趣、优先级、约束条件、从失败中学到的教训 |
| `vetoed_ideas.md` | 被明确否决的想法：日期、摘要、否决原因（除非用户主动要求复审，否则绝不再次建议） |

总量控制在约300行以内。这些文件让agent在每次新对话开始时就能立即了解：这个项目是做什么的、目前在探索哪些方向、用户的兴趣和专长是什么、哪些路已经走过不用再走。

### L2：近期记忆（Session Memory）

对应人的近期记忆，**保留最近十次对话的摘要**。每份摘要30-80行，按 `YYYY-MM-DD_NNN_session.md` 格式命名，包含：讨论了什么、做了哪些决策及理由、探索过但未采纳的想法、下一步行动、未解决的问题。

每次触发skill时，系统自动加载最近三次的对话摘要，这样agent就知道你们前几次聊了些什么，保持讨论的连续性。对于更早的L2记忆，系统维护了一个**主题索引**（Topic Index），当对话中出现匹配的关键词时，相关的历史对话会被自动召回——就像有人提到"中午吃了什么"，你自然就想起来了。

### L3：长期记忆（Cold Storage）

对应人的长期记忆。按季度归档（`YYYY-QN_archive.md`），每份约50行，只保留跨多次对话的关键主题、未被替代的决策和走过的弯路。平时只有关键词目录会被加载，只有在L2中找不到相关信息、或用户明确询问早期历史时，才会被召回。

三层记忆的加载策略总结如下：

| 层级 | 容量 | 默认加载 | 按需召回 |
|------|------|----------|----------|
| L1 | ~300行，5个文件 | 全部加载 | — |
| L2 | 10个文件，每个30-80行 | 最近3个 | 主题索引匹配时 |
| L3 | 每季度~50行 | 仅关键词目录 | 关键词匹配或用户明确请求时 |

## 记忆的流动：提升与归档

这个系统并非静态的三层存储，而是有一套**动态的记忆流转机制**，模拟人类记忆的巩固与遗忘。

### 归档：L2 → L3

当L2中的对话摘要超过10个时，最旧的一条会被清出。系统通过一个 `recall_count`（召回计数）来决定它的去向：

- **`recall_count == 0`**（从未被召回）：说明这条记忆已经不再相关，直接压缩归档到L3的季度文件中。
- **`recall_count >= 1`**（至少被召回过一次）：先评估是否有内容值得提升到L1。如果提升了，提取核心内容写入L1后从L2删除；如果不值得提升，同样归档到L3。

无论哪种情况，**该文件都会离开L2**，确保L2始终维持在10个文件以内。

### 提升：L2 → L1

每次对话结束时，系统会评估一个关键问题："这次讨论中有什么信息是在**未来每一次**对话中都需要知道的？"如果有，就将其提炼后写入L1的相应文件。例如：

- 确定了一个新的研究方向 → 添加到 `active_directions.md`
- 做了一个关键决策 → 记录到 `key_decisions.md`
- 发现用户有新的研究兴趣 → 更新 `researcher_profile.md`
- 明确否决了一个想法 → 写入 `vetoed_ideas.md`

注意：只提升**经过确认的、持久性的**信息，不提升未经验证的想法或一次性的讨论细节。

### 容量控制：L1的精简策略

L1有约300行的容量上限。当超出时，系统按优先级执行精简：

1. 移除 `key_decisions.md` 中已被新决策替代的旧条目
2. 将 `active_directions.md` 中最近10次对话都未被讨论的已完成/已放弃方向移到L3归档
3. 压缩冗余表述、合并相关条目

一旦降到上限以下即停止。这模拟了人类记忆中"重要但过时的信息逐渐被遗忘"的过程。

## 召回机制

召回是连接三层记忆的关键桥梁。系统定义了四种触发条件：

1. **主题匹配**：从当前对话中提取3-5个关键词，与主题索引比对。匹配到的L2/L3文件会被加载到上下文中（每次触发最多加载2个文件）。
2. **显式回溯**：用户说"之前讨论过"、"上次说的"等回溯性表述时，系统在召回记录的摘要中搜索匹配项并加载。
3. **L1缺乏细节**：当L1中有结论但缺少推理过程时，系统会加载记录了原始讨论的L2文件来补充上下文。
4. **矛盾检测**：当用户当前的说法与L1中的记录冲突时，系统加载相关的L2文件，同时呈现矛盾的两个来源，等用户明确确认后才更新L1，并将此次更新记录为新的关键决策。

每次召回后，系统会更新该文件的 `recall_count` 和 `last_recalled` 时间戳，并将新发现的主题关联添加到索引中。这意味着**被频繁召回的记忆不容易被遗忘**（更可能被提升到L1），而**从不被召回的记忆最终会沉入归档**（从L2移到L3）。

## 对话流程

每次触发skill后，agent会经历六个阶段：

| 阶段 | 内容 |
|------|------|
| Phase 0 | **身份加载**：读取companion的名字和人设，以角色身份贯穿整个对话 |
| Phase 1 | **记忆加载**：读取全部L1、元数据、最近3个L2；首次使用时扫描项目结构并初始化记忆 |
| Phase 2 | **综合呈现**：向用户汇报项目现状、上次讨论的内容、此后项目发生的变化 |
| Phase 3 | **方向探索**：继续已有方向、建议新方向、或通过开放式问题引导用户思考 |
| Phase 4 | **协作讨论**：一次一个问题地深入探讨，以研究框架组织思路，建设性地提出质疑 |
| Phase 5 | **收敛结论**：总结达成共识的研究问题、方法、预期成果、第一步行动和潜在风险 |
| Phase 6 | **保存记忆**：写入L2对话摘要，执行记忆的提升/归档/精简，更新元数据，以角色身份告别 |

整个过程严格限定为**纯思考模式**——只讨论想法、不写代码、不跑实验，确保agent始终作为"思考伙伴"而非"执行工具"。

## 上下文监控与优雅中断

为了应对上下文窗口耗尽的问题，系统通过hook监控上下文使用率：

- **50%**：发出警告，提示agent开始规划收尾
- **70%**：触发紧急保存——agent将当前讨论内容写入L2记忆，标记为 `Status: interrupted — context limit`

下次启动时，系统自动识别到上次是中断状态，从中断处继续讨论。这保证了即使单次对话的上下文用完，讨论的连续性也不会丢失——就像人类即使被打断，下次见面时也能靠记忆接上话题。
